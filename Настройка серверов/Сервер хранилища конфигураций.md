### Шаг 1. Создаем пользователя для хранилища
Чтобы соблюдать наш принцип изоляции («одна задача — один пользователь»), создадим отдельного юзера. Он не должен иметь доступа к базам данных 1С, только к файлам хранилища.

```bash
# Создаем пользователя 1c_repo, добавляем в нашу общую группу grp1cv8
sudo useradd -r -g grp1cv8 -s /bin/false 1c_repo
```

### Шаг 2. Подготовка папки для данных
Хранилище — это файлы. Выделим под них место (лучше не в `/home`).

```bash
# Создаем каталог для всех будущих хранилищ
sudo mkdir -p /var/1c/repos

# Отдаем права нашему новому пользователю
sudo chown -R 1c_repo:grp1cv8 /var/1c/repos

# Ставим права 770 (только владелец и группа 1С могут зайти)
sudo chmod -R 770 /var/1c/repos
```

### Шаг 3. Создаем службу systemd (Автозапуск)
В инструкции по ссылке этого нет, а это самое важное. Хранилище должно подниматься само после перезагрузки сервера.

1.  Создай файл службы:
    `sudo nano /etc/systemd/system/1c-crserver.service`

2.  Вставь туда этот конфиг (проверь версию 1С в пути `ExecStart`!):
    ```ini
    [Unit]
    Description=1C:Enterprise Configuration Repository Server
    After=network.target

    [Service]
    Type=simple
    User=1c_repo
    Group=grp1cv8
    # -d указываем папку, где лежат хранилища
    # -port указываем порт (стандарт 1542)
    ExecStart=/opt/1cv8/x86_64/crserver -d /var/1c/repos -port 1542
    Restart=always
    RestartSec=5

    [Install]
    WantedBy=multi-user.target
    ```

### Шаг 4. Запуск и проверка
```bash
# Перечитываем список служб
sudo systemctl daemon-reload

# Включаем автозапуск и стартуем сейчас
sudo systemctl enable 1c-crserver
sudo systemctl start 1c-crserver

# Проверяем, что служба "зеленая"
sudo systemctl status 1c-crserver
```

---

### Шаг 5. Как программисту создать само хранилище?
Теперь сервер работает и ждет подключений. Программист делает так:

1.  Открывает Конфигуратор своей базы.
2.  Меню: **Конфигурация -> Хранилище конфигурации -> Создать хранилище**.
3.  Путь: `tcp://IP_СЕРВЕРА:1542/НазваниеПроекта`
    *   *Например:* `tcp://10.46.101.22:1542/ERP_DEV`
4.  1С сама создаст папку `ERP_DEV` внутри твоего каталога `/var/1c/repos`.

---

### Почему это лучше, чем в твоей ссылке?

1.  **Безопасность:** Если хакер взломает хранилище, он будет "заперт" в пользователе `1c_repo`. Он даже не увидит файлы твоего основного кластера 1С, потому что у него нет прав на `/var/1c_cluster`.
2.  **Надежность:** Если процесс `crserver` упадет (например, из-за нехватки памяти), **systemd** сам поднимет его через 5 секунд (`Restart=always`).
3.  **Порядок:** Все логи хранилища и данные лежат в одном системном месте, а не разбросаны по домашним папкам.

Хранилище со временем растет. Чтобы оно не тормозило, раз в месяц проси программистов делать **«Сокращение истории хранилища»** (это делается из Конфигуратора). А ты не забывай раз в неделю делать бэкап папки `/var/1c/repos` (просто архивируй её), так как это **самые важные файлы** для разработчиков — там вся их работа за месяцы.





Это **идеальная архитектура** (Web-сервер вынесен на отдельную машину от сервера баз данных и хранилища). Так делают в Enterprise-решениях для безопасности: веб-сервер торчит в интернет, а сервер с кодом (test) спрятан глубоко в локальной сети.

### Как это работает "под капотом"
Тебе **не нужно** настраивать общие сетевые папки (Samba/NFS) между серверами! 
Веб-сервер выступит в роли прокси-переводчика. Он будет принимать HTTP-запросы от программистов снаружи, переводить их в TCP и пересылать на твой сервер `test` по порту 1542.

Вот пошаговая инструкция, как настроить это руками на твоем выделенном Web-сервере (Linux + Apache).

---

### Шаг 0. Требования к Web-серверу
На твоем отдельном веб-сервере **обязательно должна быть установлена платформа 1С** (та же самая версия, что и на сервере `test`). 
*   Сам кластер (`ragent`) на веб-сервере запускать **не нужно**.
*   Нужны только бинарники в `/opt/1cv8/x86_64/`, чтобы Apache мог взять оттуда файл модуля веб-расширения (`wsap24.so`).

---

### Шаг 1. Создаем папку публикации на Web-сервере
Зайди по SSH на твой **отдельный Web-сервер** и создай директорию, куда мы положим настройки:

```bash
sudo mkdir -p /var/www/1c_repo
sudo chown -R www-data:www-data /var/www/1c_repo
```
*(В Ubuntu/Debian пользователь Apache называется `www-data`, в РЕД ОС/CentOS — `apache`)*.

---

### Шаг 2. Создаем файл-мост (`default.vrd`)
В этой папке мы создадим конфигурационный файл 1С, который скажет Апачу: *"Все HTTP-запросы отправляй по TCP на сервер test"*.

```bash
sudo nano /var/www/1c_repo/default.vrd
```

Вставь туда этот XML-код. **Внимательно поменяй IP-адрес и имя хранилища на свои:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<point xmlns="http://v8.1c.ru/8.2/virtual-resource-system"
       xmlns:xs="http://www.w3.org/2001/XMLSchema"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       base="/repo">
    <!-- Укажи IP сервера TEST и имя папки твоего хранилища -->
    <repository connectString="tcp://10.46.X.X:1542/ИмяХранилищаНаСервереTest" allow="true"/>
</point>
```
*Сохрани и закрой файл.*

---

### Шаг 3. Настраиваем Apache на Web-сервере
Теперь нужно сказать Апачу, чтобы он реагировал на ссылку `/repo` и обрабатывал её модулем 1С.

Создаем новый конфигурационный файл в Apache:
```bash
sudo nano /etc/apache2/conf-available/1c_repo.conf
```

Вставляем туда настройки маршрутизации:
```apache
# Если на этом веб-сервере еще нет баз 1С, нужно раскомментировать строку ниже:
# LoadModule _1cws_module "/opt/1cv8/x86_64/wsap24.so"

Alias "/repo" "/var/www/1c_repo"

<Directory "/var/www/1c_repo">
    AllowOverride All
    Options None
    Require all granted
    SetHandler 1c-application
    ManagedResourceDescriptor "/var/www/1c_repo/default.vrd"
</Directory>
```

---

### Шаг 4. Включаем и перезапускаем
Применяем настройки на Web-сервере:

```bash
# Включаем наш новый конфиг
sudo a2enconf 1c_repo

# Проверяем на ошибки
sudo apache2ctl configtest

# Перезапускаем Apache
sudo systemctl restart apache2
```

---

### Шаг 5. Сетевая безопасность (Важно!)
Так как Web-сервер теперь будет стучаться к серверу `test`, убедись, что на сервере `test` (где крутится `crserver`) открыт порт 1542 для IP-адреса Web-сервера.

```bash
# Выполнить на сервере TEST:
sudo ufw allow from IP_ВЕБ_СЕРВЕРА to any port 1542 proto tcp
```

---

### Как теперь подключаются программисты?

Те, кто сидит в локалке, могут по-прежнему подключаться напрямую к `test`:
👉 `tcp://10.46.X.X:1542/ИмяХранилища`

Те, кто работает из дома (или из другой подсети), открывают Конфигуратор и вводят HTTP-адрес твоего Web-сервера:
👉 `http://IP_ВЕБ_СЕРВЕРА/repo` (или `https://...`, если ты прикрутил сертификат, как мы обсуждали ранее).

### Что сказать на собеседовании (про архитектуру):
> «Для обеспечения удаленного доступа к серверу хранилища конфигураций я развернул архитектуру с **Reverse Proxy** на базе выделенного веб-сервера Apache. 
> Вместо небезопасного монтирования файловых шар между серверами, я использовал штатный механизм веб-расширения 1С (`wsap24.so`). В дескрипторе публикации (`default.vrd`) на веб-сервере я указал параметр `<repository connectString="tcp://..."/>`. 
> Это позволило Apache принимать входящий HTTPS-трафик от удаленных разработчиков и транслировать его в TCP-сессии до внутреннего сервера `crserver`, обеспечив высокую скорость работы и скрыв топологию внутренней сети».


Раз у тебя уже настроен **HTTPS** и уже есть **опубликованные базы**, это значительно упрощает дело! Модуль 1С уже загружен в Apache, и сертификаты работают.

Тебе не нужно менять основные настройки SSL. Мы просто добавим «новую ветку» в твой веб-сервер, которая будет отвечать за хранилище.

Вот как правильно сделать Шаг 3 в твоем случае:

### 1. Используем `conf-available` (Самый чистый способ)
В Linux (Ubuntu/Debian) лучше всего создавать отдельный маленький файл для каждой задачи. Apache при запуске «склеивает» все включенные конфиги из этой папки. Поскольку базы у тебя уже есть, этот метод подхватит и HTTP, и HTTPS автоматически.

Создай файл:
```bash
sudo nano /etc/apache2/conf-available/1c_repository.conf
```

Вставь туда этот код (с учетом того, что модуль `wsap24.so` уже загружен другими базами, строку `LoadModule` здесь писать **не нужно**):

```apache
# Псевдоним: по какой ссылке будем заходить (например, /repo)
Alias "/repo" "/var/www/1c_repo"

<Directory "/var/www/1c_repo">
    AllowOverride All
    Options None
    Require all granted
    SetHandler 1c-application
    ManagedResourceDescriptor "/var/www/1c_repo/default.vrd"
</Directory>
```

### 2. Активируем конфиг
```bash
sudo a2enconf 1c_repository
sudo systemctl reload apache2
```

---

### Почему это сработает для HTTPS?
Твои файлы `000-default.conf` (порт 80) и `default-ssl.conf` (порт 443) обычно содержат директиву в самом конце (невидимо или в `apache2.conf`), которая говорит: «подключи всё, что включено в `conf-enabled`».

Так как ты сделал `a2enconf`, твой новый Alias `/repo` стал доступен **по обоим протоколам**:
*   Если кто-то зайдет на `http://твой-сервер/repo` — сработает твой редирект и перекинет на HTTPS.
*   Если кто-то зайдет на `https://твой-сервер/repo` — Apache увидит Alias, откроет папку `/var/www/1c_repo`, прочитает `default.vrd` и перешлет запрос на сервер `test`.

---

### 3. Настройка `default.vrd` (Важный нюанс)
Убедись, что в файле `/var/www/1c_repo/default.vrd` (который мы создали на прошлом шаге) правильно прописан путь к серверу, где физически живет хранилище.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<point xmlns="http://v8.1c.ru/8.2/virtual-resource-system"
       base="/repo">
    <repository connectString="tcp://10.46.X.X:1542/ИмяПапки" allow="true"/>
</point>
```

---

### Как теперь подключаться разработчику?

В Конфигураторе при подключении к хранилищу он теперь пишет **HTTPS** адрес:

👉 **`https://имя-твоего-веб-сервера.ru/repo`**

**Что произойдет под капотом:**
1.  **Конфигуратор** (ПК разработчика) стучится по HTTPS к **Веб-серверу**.
2.  **Веб-сервер** расшифровывает запрос, видит путь `/repo`.
3.  **Веб-сервер** понимает (из `default.vrd`), что это хранилище и оно живет на сервере `test`.
4.  **Веб-сервер** сам открывает TCP-соединение до **сервера `test`** по порту 1542.
5.  Данные возвращаются по цепочке обратно.

### Итог для твоего случая:
Тебе не нужно трогать файлы `000-default.conf` или `default-ssl.conf`. Просто создай папку публикации, положи туда `default.vrd` и создай новый файл в `conf-available`. Это самый безопасный способ, который ничего не сломает в уже работающих базах.

**Что сказать на собеседовании:**
> «Поскольку веб-сервер уже обслуживал HTTPS-трафик для опубликованных баз, я добавил публикацию хранилища через создание отдельного конфигурационного файла в **`conf-available`**. Это позволило использовать существующую инфраструктуру SSL-сертификатов. В дескрипторе публикации я настроил проксирование HTTP-запросов в TCP-пакеты до внутреннего сервера хранилища, обеспечив безопасный и централизованный доступ для команды разработки».
