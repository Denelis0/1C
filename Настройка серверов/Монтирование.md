Использование **systemd-юнитов** для монтирования — это самый современный и «чистый» способ в Linux (включая Astra Linux). Это превращает монтирование в полноценную службу, которой можно управлять через `systemctl`: запускать, останавливать и видеть статус.

Вот как это настроить по шагам.

### Важное правило именования
В systemd файл юнита должен называться **точно так же, как путь к папке**, но вместо слешей — дефисы.
*   Если папка `/mnt/1c_share`, то файл должен называться `mnt-1c_share.mount`.

---

### Шаг 1. Подготовка (папка и пароли)
Создаем файл, где будут логин и пароль, чтобы можно быть подключится к папке (обязательно права root у файла):
1. `sudo nano /root/.smbcredentials`
2. Впишите:
   ```text
   username=ЛОГИН
   password=ПАРОЛЬ
   domain=ДОМЕН
   ```
3. `sudo chmod 600 /etc/samba/имя_файла_с_данными`
4. Создайте саму папку: `sudo mkdir -p /mnt/1c_share`

---

### Шаг 2. Создание .mount юнита
Создайте файл описания службы монтирования:
```bash
sudo nano /etc/systemd/system/mnt-1c_share.mount
```

Вставьте туда следующий текст:
```ini
[Unit]
Description=Сетевая папка для 1С (Samba)
After=network-online.target
Wants=network-online.target

[Mount]
# Откуда берем (сетевой путь)
What=//IP_СЕРВЕРА/ИМЯ_ПАПКИ
# Куда монтируем (должно совпадать с именем файла!)
Where=/mnt/1c_share
Type=cifs
# Параметры (обязательно uid для прав 1С)
Options=credentials=/etc/samba/имя_файла_с_данными,iocharset=utf8,vers=3.0,uid=997,gid=1003,file_mode=0660,dir_mode=0770,nounix,noserverino,defaults,acl,user_xattr
TimeoutSec=30

[Install]
WantedBy=multi-user.target
```
На скриншоте — строка параметров монтирования сетевой папки (CIFS/Samba) для вашей 1С. Это «тонкая настройка», которая объясняет системе Linux, как именно работать с файлами на удаленном сервере.

Давайте разберем каждый параметр, так как они критически важны для стабильной работы 1С:

### Основные параметры:
*   **`iocharset=utf8`**: Самый важный параметр для России. Он заставляет систему использовать кодировку UTF-8, чтобы русские буквы в названиях файлов и папок 1С (например, в Журнале регистрации) отображались корректно, а не «кракозябрами».
*   **`vers=3.0`**: Указывает версию протокола SMB. Версия 3.0 — современная, быстрая и безопасная. Она хорошо работает с Windows Server 2012+ и современными NAS.
*   **`uid=997`** и **`gid=1003`**: **Это самое важное.** Поскольку в Windows и Linux разные системы пользователей, эти цифры говорят: «Считай, что владельцем всех файлов в этой папке является пользователь с ID 997 и группа 1003».
    *   *Совет:* Проверьте командой `id usr1cv8`, совпадают ли эти цифры с вашим пользователем 1С. Если у вашего `usr1cv8` другой ID, 1С может получить ошибку «Доступ запрещен».

### Права доступа:
*   **`file_mode=0660`**: Назначает права на новые файлы. `0660` означает, что владелец (1С) и группа могут читать и писать, а все остальные — не видят ничего.
*   **`dir_mode=0770`**: Назначает права на папки. `0770` позволяет пользователю 1С заходить в папки и создавать там подпапки.

### Технические параметры (решают проблемы «глюков»):
*   **`nounix`**: Отключает расширения Unix для протокола CIFS. Это нужно, когда вы подключаетесь к Windows-серверу, чтобы Linux не пытался использовать специфические функции, которые Windows не понимает. Помогает избежать ошибок с правами.
*   **`noserverino`**: Приказывает Linux самому генерировать номера файлов (inode), а не брать их с сервера. Это «лекарство» от ошибок «Слишком много открытых файлов» или когда файлы в 1С «пропадают» (stale file handle), что часто бывает при работе с сетевыми хранилищами (NAS).

### Системные флаги:
*   **`defaults`**: Набор стандартных настроек (чтение/запись, возможность запуска программ и т.д.).
*   **`acl`**: Включает поддержку расширенных прав доступа (Access Control Lists).
*   **`user_xattr`**: Позволяет хранить расширенные атрибуты файлов.

---

Это **очень хороший и грамотный набор параметров** для сервера 1С. Он настроен так, чтобы:
1.  Работали русские буквы.
2.  У пользователя 1С всегда были права на запись.
3.  Сетевое соединение было стабильным и не «отваливалось» из-за несовместимости протоколов.

**Что вам нужно проверить прямо сейчас:**
Введите в терминале: `id usr1cv8`.
*   Если вы увидите `uid=997(usr1cv8) gid=1003(grp1cv8)`, то всё **идеально**.
*   Если цифры другие — замените их в файле на свои, иначе 1С не сможет писать в эту папку.
---

### Шаг 3. Создание .automount юнита (рекомендуется)
Чтобы папка не «вешала» сервер при загрузке, если сеть еще не поднялась, создают второй файл — **автомонтер**. Он будет «будить» основную службу монтирования только тогда, когда кто-то реально обратится к папке.

```bash
sudo nano /etc/systemd/system/mnt-1c_share.automount
```

Вставьте туда:
```ini
[Unit]
Description=Автоматическое монтирование папки 1С

[Automount]
Where=/mnt/1c_share
# Тайм-аут простоя: размонтировать, если папкой не пользуются 5 минут (опционально)
IdleTimeoutSec=300

[Install]
WantedBy=multi-user.target
```

---

### Шаг 4. Запуск «сервиса» монтирования
Теперь активируйте ваши новые службы:

1. Перечитайте конфигурацию systemd:
   ```bash
   sudo systemctl daemon-reload
   ```

2. Включите **автомонтер** (именно его нужно ставить в автозагрузку):
   ```bash
   sudo systemctl enable mnt-1c_share.automount
   sudo systemctl start mnt-1c_share.automount
   ```

---

### Как теперь этим управлять?

Теперь вы можете работать с сетевой папкой как с обычной службой 1С:

*   **Проверить статус:**
    `systemctl status mnt-1c_share.mount`
    (Вы увидите, когда она была подключена и нет ли ошибок).
*   **Принудительно отключить:**
    `sudo systemctl stop mnt-1c_share.mount`
*   **Принудительно подключить:**
    `sudo systemctl start mnt-1c_share.mount`

### Почему это лучше, чем `/etc/fstab`?
1. **Зависимости:** Можно указать, что папка должна монтироваться **строго после** того, как поднимется сеть.
2. **Логирование:** Если папка отвалится, вы увидите причину в `journalctl -u mnt-1c_share.mount`.
3. **Отказоустойчивость:** Если сетевой сервер перезагрузится, `automount` сам переподключит папку при следующем обращении 1С к ней.

**Для вашей ERP на Linux это самый надежный «промышленный» вариант.** Теперь 1С сможет стабильно писать свои 80 ГБ логов или бэкапов в эту «службу».
